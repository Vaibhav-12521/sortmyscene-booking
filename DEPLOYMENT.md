# Deploying SortMyScene

Recommended setup (all free tiers), which keeps **every** feature working
including real-time seat updates and the expiry sweeper:

```
Frontend → Vercel          (static React build)
Backend  → Render          (persistent Node — WebSockets + background jobs)
Database → MongoDB Atlas    (free M0 cluster)
```

> Why not all-Vercel? Vercel runs short-lived serverless functions, which can't
> hold Socket.IO WebSocket connections or run the `setInterval` sweeper. Render
> runs a normal long-lived Node process, so it fits the backend perfectly.

First, push this project to a GitHub repo (Render and Vercel both deploy from Git).

---

## 1. MongoDB Atlas (database)

1. Create a free account at https://www.mongodb.com/cloud/atlas and create a free **M0** cluster.
2. **Database Access** → add a database user (username + password — save these).
3. **Network Access** → Add IP Address → **Allow access from anywhere** (`0.0.0.0/0`).
   (Render's outbound IPs aren't fixed on the free plan, so allow-all is simplest.)
4. **Connect → Drivers** → copy the connection string and add the DB name:
   ```
   mongodb+srv://USER:PASSWORD@cluster0.xxxxx.mongodb.net/sortmyscene?retryWrites=true&w=majority
   ```

---

## 2. Render (backend)

The repo includes **`render.yaml`**, so Render can configure the service automatically.

1. In Render → **New → Blueprint** → connect your GitHub repo → it detects `render.yaml`.
2. When prompted for environment variables, set:
   - **`MONGODB_URI`** → the Atlas string from step 1.
   - **`CLIENT_ORIGIN`** → leave **blank for now** (we'll set it after Vercel; blank = allow all origins so you can wire things up).
   - `JWT_SECRET` is auto-generated; the rest have defaults.
3. Deploy. When it's live you'll get a URL like:
   ```
   https://sortmyscene-api.onrender.com
   ```
4. Verify: open `https://sortmyscene-api.onrender.com/api/health` → should return `{"status":"ok"}`.
   On first boot it seeds the demo events (because `SEED_ON_START=true`).

> Free Render services sleep after ~15 min idle; the first request after that
> takes ~30–50s to wake. Fine for a demo.

**Manual alternative (no blueprint):** New → Web Service → Root Directory `backend`,
Build `npm install`, Start `npm start`, Health check path `/api/health`, then add the
env vars from `render.yaml` by hand.

---

## 3. Vercel (frontend)

1. In Vercel → **Add New → Project** → import the same GitHub repo.
2. **Root Directory** → set to **`frontend`** (important — the app lives in a subfolder).
   Vercel auto-detects Vite (`vercel.json` is included for SPA routing).
3. **Environment Variables** — add both (use your Render URL):
   | Name | Value |
   | --- | --- |
   | `VITE_API_BASE_URL` | `https://sortmyscene-api.onrender.com/api` |
   | `VITE_SOCKET_URL`   | `https://sortmyscene-api.onrender.com` |
4. Deploy. You'll get a URL like `https://sortmyscene.vercel.app`.

> These `VITE_*` vars are baked in at **build time**. If you change them later,
> trigger a redeploy on Vercel.

---

## 4. Lock down CORS (connect the two)

1. Back in **Render → your service → Environment**, set:
   ```
   CLIENT_ORIGIN = https://sortmyscene.vercel.app
   ```
   (your exact Vercel URL, no trailing slash). Save → it redeploys.
2. This restricts both the REST API and Socket.IO to your frontend origin.

---

## 5. Test the live app

1. Open your Vercel URL.
2. Log in with the seeded demo account: `demo@sortmyscene.test` / `password123`
   (or register a new one).
3. Pick an event → select seats → **Reserve** (watch the countdown) → **Confirm**.
4. Real-time check: open the same event on two devices/tabs and reserve a seat on one —
   it should turn amber on the other instantly.

## Environment variables reference

**Backend (Render)**

| Variable | Required | Notes |
| --- | --- | --- |
| `MONGODB_URI` | ✅ | Atlas connection string |
| `JWT_SECRET` | ✅ | Auto-generated by `render.yaml` |
| `CLIENT_ORIGIN` | recommended | Your Vercel URL (CORS + Socket.IO) |
| `RESERVATION_TTL_MINUTES` | — | Default `10` |
| `SEED_ON_START` | — | `true` seeds demo data if DB is empty |
| `PORT` | — | Injected by Render automatically — do **not** set it |

**Frontend (Vercel)**

| Variable | Required | Notes |
| --- | --- | --- |
| `VITE_API_BASE_URL` | ✅ | `<render-url>/api` |
| `VITE_SOCKET_URL` | ✅ | `<render-url>` (no `/api`) |

---

## Troubleshooting

- **CORS error in the browser console** → `CLIENT_ORIGIN` on Render doesn't exactly
  match your Vercel URL (scheme + host, no trailing slash). Fix and redeploy.
- **Seats load but realtime doesn't update** → `VITE_SOCKET_URL` is missing/wrong, or
  it has a trailing `/api`. It must be the bare backend origin.
- **First request hangs ~30s** → Render free tier cold start; subsequent requests are fast.
- **`MONGODB_URI` errors on boot** → check the password is URL-encoded and Network Access
  allows `0.0.0.0/0`.
- **Total shows ₹0** → the DB was seeded before pricing existed; redeploy with an empty DB
  (or drop the `events`/`seats` collections) so `SEED_ON_START` reseeds with prices.
